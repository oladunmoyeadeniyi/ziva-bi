## 2. Database Schema (authoritative â€” derived from ORM model files + live DB)

**Multi-tenancy mechanism:** Every operational table carries `tenant_id UUID NOT NULL FK â†’ tenants.id ON DELETE CASCADE`. The JWT embeds `tenant_id`; the `require_auth` middleware extracts and enforces it. No query in normal operation can cross tenant boundaries. Test and live are always two distinct `tenants` rows with their own `tenant_id`, so isolation is automatic â€” no table outside `tenants` itself knows or cares which environment it belongs to (M9.0.1 deliberately did not add an `environment` column anywhere else).

## 3. Architectural Invariants

| Invariant | Detail |
|---|---|
| **Cost center source of truth** | Cost centers live in `org_structure` (node_type='Cost center'). `employees.cost_center_id` and `cost_center_config.cost_center_id` both FK to `org_structure.id`. `dimension_values` may have cost_center-typed rows (legacy) but are NOT read for primary cost center lookup. |
| **Currency source of truth** | `tenant_org_config.functional_currency`, `.reporting_currency`, `.enabled_currencies JSONB` are the single authority. `tenant_fx_config` holds ONLY `fx_rates` and `revaluation_rules`. GET /api/setup/currencies reads both. |
| **Standing date floor** | No transaction date may precede `tenant_org_config.date_of_registration`. |
| **lifecycle_status gates** | CoA Replace All only when `lifecycle_status='in_implementation'`. CoA Remap only when `lifecycle_status='live'`. |
| **Expenseâ†’GL posting** | Synchronous, same DB transaction as final approval. Uncoded leaf lines (`gl_id=NULL`) block posting. Split parent containers (`is_split_parent=True`) are skipped. |
| **GL entries immutable** | Once `status='POSTED'`, journal_entry rows may not be modified. Corrections require reversing entries. |
| **CoA retired accounts** | `is_retired=True` set on remap. These rows remain in DB for historical integrity and cannot be deleted. |
| **Segregation of duties** | Period checklist: `approved_by â‰  prepared_by` enforced in router. |
| **Test-first tenant lifecycle (M9.0.1)** | Signup creates ONLY a test tenant (`environment="test"`, `parent_tenant_id=NULL`). Live is born second, only via an explicit super-admin promotion (`parent_tenant_id` then points testâ†’live, the inverse of the original M9.0 live-first/clone design â€” see `docs/BRIEF_M9_0_1_test_first_environment_flow.md`). Test stays active permanently after go-live; it is never archived. New tenants created locally before this change (e.g. the existing Red Bull pair, Â§7) still have the OLD direction (`test.parent_tenant_id` â†’ live) until the retrofit script (pending, not yet run) re-points them. |
