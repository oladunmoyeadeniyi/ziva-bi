## CONTEXT

Two fixes:

1. Custom taxes must appear inside the existing group they were assigned to,
   not in a separate "Custom" group. Only when "Create new category" is selected
   should a new group appear at the bottom.

2. The multi-level payment authorisation section must be simplified to just an
   On/Off toggle with a note pointing to Approval workflows. The inline level
   configuration fields (role inputs, min amount inputs, + Add level button) must
   be removed entirely. Approver roles and thresholds are configured in Approval
   workflows where employee and role data is available.

{/* New custom categories only */}
{(() => {
  const countryCode2 = (org.country && TAX_PROFILES[org.country]) ? org.country : "XX";
  const existingGroupTitles = new Set(
    (TAX_PROFILES[countryCode2] ?? TAX_PROFILES["XX"]).map(g => g.title)

  const newCategoryItems = config.tax_items.filter(
    t => t.custom && t.category && !existingGroupTitles.has(t.category)

  const byCategory = newCategoryItems.reduce((acc, item) => {

    if (!acc[cat]) acc[cat] = [];
    acc[cat].push(item);
    return acc;
  }, {} as Record<string, TaxItem[]>);

    const isCollapsed = collapsedTaxGroups.has(`custom_${category}`);
    return (

## NOTES FOR CC

- Custom taxes assigned to an EXISTING group title (e.g. "Indirect taxes",
  "Direct taxes", "Employment & payroll") must render INSIDE that group,
  after the jurisdiction items, with a light blue background to distinguish them.
- Custom taxes assigned to a NEW category (not matching any existing group title)
  render in their own separate blue-bordered group below all jurisdiction groups.
- The group header count must include custom items in its total.
- The multi-level auth toggle stays — only the inline configuration fields are
  removed. The `use_multilevel_auth` boolean in `OrgConfiguration` is preserved.
- Do not remove `auth_levels` from the backend schema — it may still be used
  in future. Only remove it from the frontend interface and default config.
